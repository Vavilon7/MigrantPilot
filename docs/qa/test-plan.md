# Тест-план. Часть 1: стартовая страница, регистрация, авторизация, работа с письмами

## 1. Общая информация

**Проект:** многофункциональная AI-платформа помощи мигрантам в Германии.

**Фокус первой части тестирования:** базовый путь пользователя от первого открытия приложения до работы с письмом от ведомства.

**Основание:** документ "Проект приложения.pdf" и предоставленные макеты интерфейса.

**Ключевая идея проекта:** приложение помогает мигрантам понимать немецкую бюрократию простым языком. На первом этапе особенно важны письма от ведомств, AI-помощник, налоги и базовый профиль.

**Самая важная функция для MVP:** AI-анализ писем от Jobcenter, Finanzamt, Familienkasse, Ausländerbehörde и других ведомств.

## 2. Цели тестирования первой части

- Проверить, что новый пользователь понимает назначение приложения с начальной страницы.
- Проверить, что пользователь может перейти со стартовой страницы к регистрации или авторизации.
- Проверить регистрацию с валидными и невалидными данными.
- Проверить авторизацию существующего пользователя.
- Проверить выбор языка интерфейса.
- Проверить базовую навигацию после входа.
- Проверить загрузку письма от ведомства.
- Проверить анализ письма и отображение результата простым языком.
- Проверить, что пользователь видит сроки, риски, необходимые документы, готовый ответ, цену, возможность оплаты и почтовый адрес для отправки.
- Проверить дисклеймеры, GDPR-сообщения и защиту персональных данных.
- Проверить, что приложение позиционируется как информационный сервис, а не как юридическая консультация.

## 3. Объём тестирования

### 3.1. Входит в объём

- Стартовая страница.
- Онбординг и объяснение ценности приложения.
- Регистрация.
- Авторизация.

- Проверка предупреждений о сроках и рисках.

- Пользователь понимает назначение приложения за первые секунды.
- Основной путь "Начать сейчас" работает.
- Альтернативный путь "Войти" работает.
- Сообщение о защите данных отображается до регистрации.
- Нет критичных визуальных дефектов.

### 11.9. Безопасность и GDPR для писем

- Загруженное письмо доступно только владельцу аккаунта.
- Пользователь не может открыть чужое письмо по прямой ссылке.
- Документы передаются только по защищённому соединению в production-среде.
- Пользователь видит информацию о защите данных перед загрузкой.
- Если включено автоудаление документов, файл удаляется согласно выбранному режиму хранения.
- Персональные данные из письма не отображаются в логах интерфейса.
- При ошибке анализа приложение не показывает технические детали backend/API.

### 11.10. Критерии приёмки блока

- Пользователь может загрузить письмо в поддерживаемом формате.
- Пользователь получает результат анализа простым языком.
- Система показывает срок ответа или честно сообщает, что срок не найден.
- Система не придумывает данные при плохом качестве документа.
- Пользователь видит риски, цену готового ответа и почтовый адрес ведомства.
- Пользователь может оплатить и скачать готовый файл ответа.
- Пользователь видит дисклеймер об информационном характере помощи.
- В интерфейсе нет юридических гарантий результата.
- Документы пользователя защищены от доступа других пользователей.

## 13. Риски для первой части

| Риск | Влияние | Что проверять особенно внимательно |
| --- | --- | --- |
| AI придумывает срок или требование | Пользователь может пропустить важный срок или отправить неверный ответ | Проверять письма без срока, плохое качество, разные ведомства. |
| Плохое OCR-распознавание | Пользователь получает неверный анализ | Проверять фото, сканы, низкое качество, частично обрезанные письма. |
| Неточный юридический тон | Пользователь может принять подсказку за юридическую консультацию | Проверять дисклеймеры и формулировки результата. |
| Гарантирующие формулировки | Пользователь может ожидать юридически гарантированный результат | Проверять тексты на отсутствие обещаний "гарантируем", "обеспечим", "решим вопрос". |
| Утечка персональных данных | Критический GDPR-риск | Проверять доступ к документам, прямые ссылки, сессии, удаление. |
| Слишком сложный интерфейс | Пользователь с низким немецким не поймёт, что делать | Проверять тексты, подсказки, локализацию, короткие сценарии. |
| Слишком много способов связи с ведомством | Пользователь может запутаться или выбрать неподходящий канал | Основной путь должен быть один: оплатить, скачать ответ и отправить по почте. |
| Ошибка регистрации или входа | Пользователь не дойдёт до основной ценности продукта | Проверять P0-сценарии авторизации в каждой сборке. |
